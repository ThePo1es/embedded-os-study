# OSTEP 4 - 6 : 시스템 콜 vs SMC 오버헤드 계측

## 1. 개요 및 측정 결과

본 실습은 사용자 모드에서 커널 모드로 진입하는 **시스템 콜(System Call)**의 오버헤드를 측정하고, 이를 **SMC (Secure Monitor Call)**의 오버헤드 특성과 비교 분석하는 데 목적이 있다.

- **측정 환경:** 리눅스 (WSL/Ubuntu 등)
- **측정 대상:** `getpid()` (단순 시스템 콜), `read()` (파일 디스크립터 검증 등의 추가 작업이 포함된 시스템 콜)
- **측정 방법:** 1,000,000회 반복 호출 루프의 소요 시간을 측정한 뒤 이를 반복 횟수로 나누어 **1회당 평균 비용(us)** 도출 (최소 5회 반복 측정)

### 📊 측정 결과 (예시)

```text
=== 평균 오버헤드 결과 (5회 평균) ===
1. getpid() [gettimeofday 기점]:  0.07253 us
2. getpid() [clock_gettime 기점]: 0.06869 us
3. read()   [clock_gettime 기점]: 0.07767 us
```

## 2. 결과 분석 및 비교

### 🔍 `gettimeofday()` vs `clock_gettime()`

*   **`gettimeofday()`**: 전통적인 UNIX 시간 측정 방식. 반환되는 해상도가 마이크로초(us) 단위이며 시스템 시스템 전역의 Wall Clock Time을 측정한다.
*   **`clock_gettime()`**: `CLOCK_MONOTONIC`을 사용할 경우 시스템 부팅 이후의 단조 증가하는 시간을 나노초(ns) 해상도로 반환한다. 
*   **성능 차이**: 최근 리눅스 환경에서는 두 함수 모두 **vDSO (Virtual Dynamically Shared Object)** 영역을 통해 유저 스페이스에서 직접 처리되는 경우가 많아 커널 진입이 생략되지만, 데이터 구조 크기와 연산 해상도 차이로 인해 보통 `clock_gettime`이 정밀도와 속도 측면에서 더 유리하거나 유사한 결과를 보인다. 

### 🔍 `getpid()` vs `read()`

*   **`getpid()`**: 호출 프로세스의 PID만 반환하는 전형적인 **Null 시스템 콜(가장 가벼운 시스템 콜)**. 인자 변환이나 권한 검사, 리소스 접근이 불필요하여 순수 Context Switch 비용에 가장 근접한다. (결과: ~0.068 us)
*   **`read()`**: fd(`/dev/null` 등)의 유효성 검사, 커널 내부 버퍼 구조체 접근 등의 추가적인 VFS(Virtual File System) 계층 로직이 포함된다. 따라서 `getpid()`보다 평균 처리 비용이 약간 더 크게 나타난다. (결과: ~0.077 us)

## 3. 심화 분석 (생각해보기)

### 🤔 1. 왜 반복 루프 안에서의 호출이 '완전한 syscall 비용'이 아닐 수도 있는가?

반복 루프를 이용해서 시간을 재면 `(전체 측정 시간) / 1,000,000` 방식으로 계산한다. 이 시간 안에는 다음과 같은 요소들이 함께 포함되어 있다.
*   루핑(Loop) 오버헤드: `for` 문의 `i++` 연산과 조건 검사, 분기(branch) 명령어 실행 비용
*   함수 호출 오버헤드: C 라이브러리 (glibc) 래퍼 함수를 호출하는 데 따르는 약간의 오버헤드
즉, 커널 경계를 넘는 "순수한 트랩(Trap) 및 모드 스위치" 시간뿐만 아니라 부가적인 유저 스페이스 명령 오버헤드가 섞여 실제보다 약간 과대평가될 수 있다. 

### 🤔 2. 캐시, 파이프라인, CPU 주파수 변동(DVFS)의 영향은?

*   **캐시(Cache) 및 TLB (Translation Lookaside Buffer)**: 모드 전환이 일어날 때 캐시 무효화나 TLB 플러시가 발생할 수 있다. 100만 번 같은 코드를 반복하면 **Instruction/Data 캐시 적중률(Cache hit rate)이 극단적으로 높아지게 되어** 실제 워크로드 환경보다 더 빠르게 측정(과소평가)되는 결과를 낳을 수 있다.
*   **파이프라인 플러시**: Trap(예외)이 발생해 분기할 때 파이프라인 안의 명령어들이 비워지며(flush), 이 복구 과정이 오버헤드에 포함된다.
*   **CPU 주파수 동적 스케일링(DVFS)**: 루프가 수백만 번 도는 동안 CPU 점유율이 100%로 치솟으면서 스레드가 부스트 클럭(Boost Clock) 상태로 진입하게 된다. 따라서 1회 측정과 백만 번 연속 반복 측정 간에는 실행 속도의 차이가 크다.

### 🤔 3. 유저 모드 ↔ 커널 모드 전환 시 어떤 일이 일어나는가?

제한적 직접 실행 (Limited Direct Execution) 하에서 프로그램이 운영체제에 통제권을 넘기기 위해 일어나는 스위칭 과정이다.
1.  **레지스터 및 PC 저장**: 사용자 모드의 범용 레지스터, 스택 포인터(SP), 프로그램 카운터(PC) 등을 각 프로세스의 커널 스택(Kernel Stack)에 저장한다.
2.  **모드 비트 변경**: 하드웨어 레벨에서 권한 레벨이 승격된다 (e.g. Ring 3 ➙ Ring 0).
3.  **트랩 테이블 참조**: 하드웨어가 시스템 콜 테이블을 참조하여 해당하는 커널 핸들러로 분기점(Jump)을 이동시킨다.
4.  작업 완료 후 `return-from-trap` 명령어로 저장된 레지스터 상태를 복구하고 다시 권한을 하향하여 사용자 프로그램으로 돌아온다.

### 🤔 4. TrustZone의 SMC는 왜 시스템 콜보다 더 비쌀 것으로 예상되는가?

*   **더 많은 상태 (State) 저장 및 복원 요구**: 시스템 콜은 User ↔ Kernel(Supervisor) 간의 전환이지만, SMC는 Normal World에서 **Secure World (EL3 / Secure Monitor)** 로 진입하는 과정이다. 이때 General Purpose 레지스터 외에도 시스템의 핵심 Control/Banked 레지스터 영역들의 값까지 전환해야 하므로 Context Switch 범위가 더 넓다.
*   **캐시/TLB 엄격한 격리 및 재적재**: Normal World와 Secure World는 메모리를 엄격히 구분하므로, 세계가 전환될 때 캐시나 TLB의 공유나 Flush 동작이 커널-유저 모드 전환 시보다 더 큰 패널티로 작용한다. (Cold Cache 현상 심화)

---

### 📝 결론
시스템 콜은 단순 함수 호출보다 수 배 이상 비싸지만 (대략 수십~수백 나노초 이상), 이는 안전한 하드웨어/자원 접근 제어와 '제한적 직접 실행'을 위해 반드시 지불해야 하는 보안 및 관리 비용임을 확인할 수 있었다. 이를 확장하여 TrustZone 접근 시 발생하는 SMC 비용 역시 설계 시 주요한 오버헤드 고려 요소이다.
